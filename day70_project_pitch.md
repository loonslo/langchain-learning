# 5 分钟项目讲解（Day70）

> 用途：面试时被问"讲讲你这个项目"。目标不是报菜名，是让面试官记住一句话——**"这人会做能测、能回归、能上线的 LLM 系统，不是只会调 API。"** 计时讲，讲完留问题给对方追。

## 0:00–0:30 一句话定位

我做了一个**企业知识库 Agent + 自动化评测平台**。普通候选人交一个聊天框 demo，我交的是一个带**评测门禁、能回归、能上线**的系统——这正好是我测试工程背景的延伸：把不稳定的模型能力，放进可控的工程流程里。

## 0:30–1:30 它能干什么（数据 → 问答 → 权限）

- 数据侧：解析 PDF/Markdown，混合检索（向量 + BM25）解决专有名词召回不准，Chroma 持久化，答案带来源溯源。
- 真实数据不是建一次就完——做了**增量同步**，文档增删改只处理变化的，不全量重建。
- 企业最痛的**文档级权限**：检索层按用户身份过滤，A 部门问不出 B 部门的机密——而且是在检索层做，不靠 prompt（靠 prompt 一次注入就泄露）。
- 多租户隔离 + JWT 认证 + 限流，无 token 调不通，跨租户查不到数据。

## 1:30–3:00 护城河：评测 + CI 门禁（重点讲这段）

- 我把评测做成独立作品，不是"跑一次 RAGAS 出个数"：50+ 条评测集覆盖事实 / 跨文档 / **拒答** / 引用准确性，评测集进 git 当回归用例库。
- 关键数字（讲你自己跑出来的真实值）：拒答正确率 ___%、关键词命中率 ___%，改了 chunk 策略后从 ___% 到 ___%——每个数都有计算方法，不是拍的。
- 最硬的一点：**CI 评测门禁**。每个 PR 自动跑 pytest + 评测，幻觉率 / 拒答召回率跌破阈值，GitHub Actions 直接红灯、拦 merge。我可以现场演示：故意删掉 prompt 里的拒答约束，push 一个 PR，CI 就红。
- 这就是测试背景的差异化：评测集 = agentic loop 的反馈信号，质量门禁挡在合并之前。

## 3:00–4:00 上线与可观测（讲 tradeoff）

- FastAPI 服务化、Docker 打包、部署到云主机拿到公网地址，手机能访问（线上用可公开语料，真实脏数据只在本地跑——这是合规边界）。
- 可观测：trace 定位单次失败是检索、生成还是权限问题；监控看 p95/p99 + 错误率 + 成本趋势，不看平均值。
- 成本×质量 tradeoff：加缓存 + model routing 后，省了 ___% 成本、质量掉 ___%（用评测集验证没塌）。这种"省钱但守住质量"的取舍，是我会主动量化的。

## 4:00–5:00 收尾 + tradeoff 自评

一句话收：**模型是 agent 系统里最小的部分，决定生产可靠性的是 harness——工具调用、记忆、可靠性、安全、评测。我做的就是 harness。**

主动抛 1-2 个我做过取舍的点，引面试官深挖：
- 权限为什么放检索层不放 prompt；
- 为什么评测集既当报告又当 pytest 回归既当 CI 门禁（一份来源、三处复用）；
- Chroma vs pgvector 我现在选 Chroma 的理由，以及上量后怎么换。

> 讲解纪律：结论先行、用数字、讲 tradeoff；不吹"懂底层原理"，不报一堆名词。每句话都能指到仓库里的一个文件。

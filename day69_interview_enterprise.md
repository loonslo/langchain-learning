# 面试题准备 · 企业级与上线（Day69）

> 接 `面试题准备.md`（RAG/Agent/选型）。这一份补企业第一梯队和上线侧——大厂面 AI 应用岗，问题八成落在这里。每题要能讲清"为什么这么做"，并指向项目里的具体代码。

## 一、可靠性与成本

**Q：模型 API 超时 / 限流怎么办？**
超时设上限 + 指数退避重试；多供应商时主备 fallback（主挂了切备）；客户端限流保护下游。重试要幂等，避免重复扣费 / 重复副作用。对应 `day36_reliability.py`。

**Q：怎么在不掉质量的前提下降本？**
三招叠加：① 缓存（精确 + 语义缓存，重复问不再调模型）；② model routing，简单问题走便宜模型、难的才上贵的；③ prompt 压缩 / 上下文裁剪减 token。关键是每一步都用评测集回归，确认质量没塌——能讲"省 X% 成本、质量掉 Y%"这组数才算数。对应 `day37` + 阶段2 成本基线。

**Q：成本怎么监控？**
每次请求记 token 和成本落库，按天聚合看趋势。成本悄悄涨往往是缓存命中率掉了或路由失效。对应 `capstone/monitoring.py`。

## 二、文档级权限与多租户

**Q：文档级权限为什么必须在检索层做，不能靠 prompt？**
靠 prompt = 把机密塞进上下文再求模型别说，一次注入或一次模型抽风就泄露。正确做法是检索时就按用户身份过滤，无权的 chunk 根本进不了上下文。对应 `capstone/permissions.py`，`can_see()` 在召回后、进 prompt 前就把无权内容滤掉。

**Q：多租户怎么隔离，才不会串数据？**
"查询时加 tenant 条件"是软隔离，写漏一行就串。强隔离是物理隔离——每租户独立 collection / 独立 schema / 独立向量库目录，查不到别人的。对应 `auth.tenant_chroma_dir()`。

**Q：认证怎么做？**
JWT 或 API key。登录签发 token，身份和租户写进 claims，每次请求带着；无 token / 过期 → 401。限流按租户滑动窗口，超频 429。对应 `capstone/auth.py`。

## 三、CI 评测门禁与质量回归

**Q：prompt 改了怎么保证没改坏别的？**
评测集进 git 当回归用例库，每次改跑全集，画指标随 commit 的回归曲线，一眼看出哪次改动让幻觉率涨了。再把它接进 CI：评测分数跌破阈值就让 PR 失败。对应 `capstone/ci_gate.py` + `.github/workflows/eval-gate.yml`。

**Q：你怎么证明 CI 门禁真的有用？**
故意改坏一个 prompt（比如删掉拒答约束），push 一个 PR，CI 评测红、merge 被挡——这就是演示。这是"测试转 AI"最硬的作品点。

## 四、上线补全

**Q：监控该看什么？**
不是平均值，是分位数（p95/p99）+ 错误率 + 成本趋势。长尾慢请求才是用户骂街的原因，平均值会把它掩盖。越线再告警，避免告警疲劳。对应 `day62`。

**Q：生产为什么不裸用 Chroma？**
Chroma 适合原型；生产常换 pgvector（向量和业务同库、能 SQL 过滤、复用 Postgres 运维）或 Milvus（亿级 / 高并发）。对应 `day63`。

**Q：国内上线还要做什么（合规）？**
输入输出双向内容安全过滤 + 审计留痕；生成式服务面向公众要做算法 / 大模型备案。技术跑通只是一半。对应 `day64`。

**Q：企业为什么常用 Azure OpenAI？**
数据驻留自己租户、走企业合同和 SLA、合规过得去。所以模型调用要抽一层，改一个环境变量就能切供应商。对应 `day65_provider_abstraction.py`。

**Q：系统大概扛多少并发？瓶颈在哪？**
locust 逐步加压，看 QPS 和 p95 的拐点。LLM 应用瓶颈通常不在自己的 FastAPI，而在上游模型 API 限流和单次推理耗时——压测让你拿数据说话。对应 `day66`。

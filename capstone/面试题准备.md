# 面试题准备（Day54）

> 按"能口头讲清 + 能拿代码/数字佐证"标准准备。每题给要点，不要背稿。

## 一、RAG 核心

**Q：RAG 为什么会答错？怎么判断是检索错还是生成错？**
两类：检索没召回相关内容 vs 召回了但生成乱编。判断：看召回的上下文里有没有相关信息——没有就是检索问题（调 chunk / 混合检索 / 查询改写 / reranker），有但答跑偏就是生成问题（调 prompt / 加忠实度约束 / 换模型）。LangSmith trace 能直接看到每步。

**Q：chunk_size 怎么选？不同文档怎么处理？**
没有万能值。太小断意、太大带噪声。中文起点 300–500、overlap 取 10–20%，再用评测的 context_precision 调。表格/代码类文档要按结构切，不能纯按长度。

**Q：怎么构造 RAG 测试集？**
按题型覆盖：事实、跨段落、拒答、引用准确性。每条 = 问题 + 标准答案 + 关键词 + 应命中来源 + 类型。存 JSON 做版本管理，给 pytest / RAGAS / LangSmith 共用。拒答题最关键——直接量化防幻觉。

**Q：混合检索解决什么？**
向量懂语义但对专有名词 / 编号 / 型号不敏感，BM25 关键词召回补上，EnsembleRetriever 按权重融合。领域里缩写编号多就明显更好。

## 二、Agent / LangGraph

**Q：LangGraph vs Chain？**
Chain（LCEL）是固定直链；LangGraph 用图描述带状态的流程，支持循环、分支、状态持久化、中断恢复——Agent 必需的能力 Chain 给不了。

**Q：状态怎么持久化？怎么防死循环？**
checkpointer（InMemorySaver / SqliteSaver）按 thread_id 存每步快照，多轮记忆 + 中断恢复都靠它。防死循环用 recursion_limit，超步数抛 GraphRecursionError。

**Q：高风险工具怎么 HITL？**
执行前 interrupt() 把图暂停、把动作抛给人，Command(resume=...) 恢复；依赖 checkpointer。且危险工具只模拟，不给真实删除 / 发送 / 支付权限。

**Q：ReAct 和 Plan-and-Execute 区别？**
ReAct 走一步看一步（想→做→看→再想），灵活但步数多易跑偏；Plan-Execute 先规划全程再执行，可控、省 LLM 往返，但计划错了全错。

**Q：MCP 解决什么？**
工具 / 数据源的统一协议（"插座"）。工具方按 MCP 暴露，Agent 端一个客户端即插即用，换工具不改 Agent 代码，跨团队跨语言复用。

## 三、选型与工程

**Q：RAG vs 微调怎么选？**
知识常更新 / 要溯源 / 成本敏感 → RAG；要固定风格术语且知识静态、预算够 → 微调；需求简单先 Prompt。能 Prompt 不 RAG，能 RAG 不微调。

**Q：怎么降低幻觉？**
检索侧提召回质量（混合检索 / reranker / 查询改写）；生成侧 prompt 约束"只依据上下文 + 没有就拒答"；评测侧用忠实度 + 拒答正确率持续盯。

**Q：prompt 改了怎么回归？线上幻觉率 / 成本怎么监控？**
prompt 当版本管理，每次改完跑评测集回归（pytest），对比指标。线上：结构化日志记 token / 延迟 / 错误，按拒答率 / 忠实度抽样评估，超阈值告警。

## 四、反问与定位（收尾）

主动亮护城河：测试背景 → 做"可评测、可回归"的 Agent 系统（harness engineering）。强调有完整"评测集 + 指标 + 报告 + 失败用例库 + 回归"闭环，能拿出带计算方法的数字，而不是"感觉还行"。
